# 第1週: 企画とコンセプト立案

## 〜 「誰の、どんな朝」を救うか決めよう 〜

---

## 1. コンテストの概要

今回の目標は、**「毎朝見たいwebダッシュボード」**をチームで開発することです。

- **期間**: 4週間（企画 → 開発 → ブラッシュアップ → 発表）
- **技術スタック**: React + Vite + Tailwind CSS
- **評価**: 最終日の成果発表会にて、50点満点で採点

---

## 2. 「作るもの」を外さないコツ

良いアプリを作るには、**機能（API）から考えてはいけません。**

### ① 課題定義から入る
「天気APIが使いたいから天気アプリを作る」はNGです。
「**朝、傘を忘れて駅で絶望する人を救いたい**」という課題（ペルソナの悩み）から始めましょう。

### ② ペルソナを「徹底的に」具体化する
「大学生」ではなく「**朝8時15分のバスに乗らないと1限に遅刻する、ギリギリまで寝ていたい一人暮らしの大学生（名前：佐藤くん）**」くらい具体的にしましょう。

> **コツ**: その人が朝起きてから家を出るまでの「感情の動き」を想像してみてください。

---

## 3. 採点項目と高得点の秘訣

審査員（メンター）はここを見ています。

### 1. APIの利用 (20点) 🌟 最大配点
**コツ**: 単に数字を出すだけでは不十分。「降水確率80%」ではなく「**今日は絶対傘を持ってけ！**」と表示するような、ユーザーの行動に繋がる「意味付け」をすると加点されます。

### 2. 開発体制 (10点)
**コツ**: Gitのコミットログが一人だけになっていませんか？ 
「ここのCSS直したよ」「API繋げたよ」といった、協力の跡が見えるコミットメッセージやブランチ運用が評価されます。

---

### 3. デザイン (10点)
**コツ**: 凝った装飾より「**情報の優先順位**」です。
忙しい朝、一番最初に見るべき情報は何か？ それを最大級の文字で表示し、不要な情報は削ぎ落とす「引き算のデザイン」を目指してください。

### 4. 使いたいか (10点)
**コツ**: 結局は「愛」です。
プレゼンで「これ、自分でも明日から使いたいです！」と自信を持って言えるくらい、特定の個人の悩みを解決しきっているか。
**「万人受け」を狙わず「一人に深く刺さる」**ものを作った方が、結果的に評価は高くなります。

---

## 本日のワーク

1. **チームビルディング**: チーム名と役割分担を決めよう
2. **課題の深掘り**: チームメンバーの「朝の悩み」を共有しよう
3. **ペルソナ設定**: 「誰を救うか」を1人に絞り込もう
4. **要件定義シートの作成**: `docs/requirements-sheet.md` を埋めよう
